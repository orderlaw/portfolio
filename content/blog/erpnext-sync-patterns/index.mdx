Connecting ERPNext to external systems is one of the most common things I do, and one of the most consistently underestimated. On the surface it looks simple — read data from here, write it there. In practice, the failure modes are subtle and they compound.

Here are the patterns I keep coming back to, and the ones I've stopped using after they broke.

---

## The patterns that work

### Event-driven over polling

Don't poll ERPNext for changes on a schedule. Use document hooks — `on_submit`, `on_update`, `after_insert` — to fire events when something actually happens. Polling creates drift: you check every 5 minutes, but the important thing happened 4 minutes and 50 seconds ago. Event-driven means the sync happens when the data changes, not when your cron happens to run.

ERPNext's webhook system is reliable enough for most production use. For anything that can't afford to miss an event, pair the webhook with a nightly reconciliation pass that checks for anything that slipped through.

### Idempotent writes

Every write operation you make to an external system should be safe to run twice. External systems go down, n8n executions get retried, webhooks get delivered more than once. If your write creates a duplicate on the second call, you have a problem.

Most APIs have a way to handle this — an `idempotency-key` header, an upsert endpoint, a natural unique key you can check against before writing. Find it and use it before anything else.

### Translate, don't map

ERPNext has its own data model. The external system has a different one. Don't try to keep them in sync field-for-field — build a translation layer that explicitly converts between the two representations. When one side changes, you update the translation layer, not every workflow that touches it.

This adds indirection upfront. It saves hours when a field gets renamed or a new status gets added mid-project (and it always does).

### Separate sync from validation

Don't put business logic inside the sync workflow. The sync should do one thing: move data. Validation — checking whether the data makes sense, whether required fields are present, whether the values are in range — should happen before the sync runs. If the data is bad, fail loudly and early, not silently mid-workflow.

---

## The patterns I've stopped using

### Direct database access

It's tempting to query ERPNext's MariaDB directly for read operations, especially when the REST API is slow or doesn't expose the fields you need. Don't. ERPNext's data model is not stable across upgrades. Field names change, table structures get reorganised, calculated fields move. The REST API is a contract. The database is not.

### Synchronous multi-step workflows

If your sync workflow has five steps that all need to complete before it's "done," and step three sometimes takes 30 seconds, you're going to hit timeouts. Break long chains into smaller async workflows that hand off via a queue. Each step completes independently, and failures don't cascade.

### Storing state in n8n

n8n's execution history is not a database. Using it to track "last synced ID" or "pending items" works until it doesn't — executions get pruned, the history is not easily queryable, and it becomes a debugging nightmare. Store sync state in a proper database. Supabase works well for this.

---

## One thing that matters more than patterns

Most sync projects fail slowly. The workflow runs fine for two weeks, then an edge case appears, then another, then the client stops trusting the data, then they go back to doing it manually. The workflow technically still runs.

Build the reconciliation check before you deploy, not after something goes wrong. A daily job that compares record counts and flags discrepancies is boring to build and genuinely valuable. It's the difference between knowing something's wrong and finding out a week later.
